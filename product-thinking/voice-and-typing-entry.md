# 语音与打字：把“语音入口”做成输入法的第一等能力（我的主见）

> 编码：UTF-8  
> 创建时间：2026-03-20T21:15:00+08:00  
> 关注平台：Windows + Android（当前优先 Android）  
> 关联研究：`TODO/ime-research/notes/20260320_voice_typing_keyboards.md`  

## 1. 我对“语音 vs 打字”的判断

- **语音是“高吞吐输入”**：适合长句、叙述、开会速记、驾车/走路等场景；单位时间产出更多文本。
- **打字是“高精度编辑”**：适合纠错、结构化输入（邮箱/密码/代码/表格）、私密输入与静音场景。
- 因此，**真正可用的输入法一定同时把两者做到“互补”**：语音负责产出，打字负责控制与修正。

一个常见失败模式是：把语音做成“大按钮 + 输出文本”，但没有把“纠错/编辑/撤销/断句/标点/数字”等环节与键盘能力打通，导致用户体验在 30 秒后崩塌。

## 2. 三种产品形态（以及我更推荐哪一种）

### 形态 A：键盘内置语音听写（单 APK 一体化）

**优点**：体验完整，语音模式与键盘编辑无缝；能做更强的“边说边改”。  
**代价**：体积、耗电、性能、崩溃面、合规模型分发都会急剧上升。  

开源/可源码参考的代表：
- AnySoftKeyboard：在键盘里触发系统语音识别 Intent（质量取决于系统提供者）。
- FUTO Keyboard：在键盘里集成离线 Whisper/ggml（但许可为 source-available，使用/分发需重点评估）。

### 形态 B：打字键盘 + 独立语音 IME（组合式，“看起来一体”）

打字键盘提供一个“语音快捷键/动作”，按下即切到语音 IME；语音结束自动切回打字键盘。

**优点**：强隔离（语音崩了不拖累键盘）、实现可插拔、合规边界清晰；非常适合从 0→1。  
**缺点**：要把“自动切回、文本插入、撤销、光标同步”做得很稳，否则会有割裂感。  

代表：
- HeliBoard / FlorisBoard / OpenBoard（打字键盘侧做切换入口）
- whisperIME / FUTO Voice Input / Sayboard（语音 IME 侧做听写）

### 形态 C：非 IME 的语音听写工具（Windows 上尤其现实）

语音识别作为独立应用运行，把文本注入到当前焦点窗口（模拟键盘输入或调用更底层 API）。  
**优点**：绕过 TSF/COM 的复杂安装链路，工程风险低。  
**缺点**：与 IME 的候选/编辑深度融合更难。

## 3. 我建议的路线（针对你当前的优先级：Windows + Android）

### Android：先做“组合式”打通闭环，再决定是否内置离线语音

1. **先选一个打字键盘基座**（自研或基于开源）：先把拼音/词典/候选/皮肤/设置做稳。
2. **语音入口先走形态 B**：你可以先不写 ASR，只做“切换到语音 IME + 自动切回 + 插入后的编辑策略”。
3. **离线语音单独做成一个模块/进程**：如果后续要内置 Whisper，尽量让语音在独立进程里跑，键盘主线程只收“partial/final text events”。
4. **最后再考虑形态 A**：当且仅当你能证明“语音与键盘编辑的耦合”能带来明显留存提升，并且能承受体积/电量/合规成本。

### Windows：把打字 IME 与语音听写解耦，先跑通可用性

- 打字 IME：优先复用成熟引擎（如 Rime），把用户词库/云同步/皮肤等外围能力做起来。
- 语音：先用独立听写工具（形态 C），等语音模型与 UX 稳定后，再评估是否值得深度集成到 TSF。

## 4. 最大挑战（我认为真正决定成败的点）

1. **纠错闭环**：语音识别永远会错；如果“修改成本”不低，用户不会持续用。
2. **首字延迟与稳定性**：语音入口必须给出“马上有反馈”的感觉；并且不能把 IME 卡死。
3. **标点/断句/格式化**：中文标点、数字、日期、专有名词、代码片段、英文缩写，都是高频痛点。
4. **混合输入策略**：语音输出后立刻用键盘编辑，需要处理光标、选区、撤销栈、粘贴板、候选学习等。
5. **模型与电量**（离线语音时）：模型体积、运行时峰值功耗、发热与后台限制，都会影响真实可用性。
6. **许可与分发**：很多“看起来能用”的源码项目，其实许可/模型权利/分发条款并不允许你按商业产品方式使用。

## 5. AI 大模型在这个领域容易犯的错（提醒自己保持主见）

- 低估平台差异：把 Windows TSF 与 Android IME 当成“差不多”是错的。
- 过度承诺离线语音：在真实机型上做到低延迟/低耗电/高准确比 demo 难一个数量级。
- 忽略许可：把 source-available 当成开源，或者忽略模型权利，会在后期爆雷。
- 忽略工程边界：把语音/LLM/词典/皮肤全塞进一个进程，最终只会变成卡顿与崩溃。

## 6. 下一步我建议你立刻做的事（最小闭环）

1. 明确你的“主场景”与“非主场景”（例如：聊天长句 vs 表格/密码）。
2. 做一个最小可用原型：打字键盘 + 语音 IME 切换 + 自动切回 + 基本纠错手势。
3. 定义可量化指标：首字延迟、每分钟崩溃率、电量、用户主动纠错次数、最终保留率。
