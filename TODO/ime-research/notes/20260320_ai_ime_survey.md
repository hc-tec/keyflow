# AI 输入法/键盘：开源方案调研（Windows + Android 视角）

> 编码：UTF-8  
> 创建时间：2026-03-20T18:15:00+08:00  
> 目标：回答“有没有现成的 AI 输入法方案？它们怎么做的？落地最大挑战是什么？”并给出我自己的判断  
> 说明：本笔记优先收录我已拉取到本地的仓库（见 `TODO/ime-research/notes/20260320_repo_inventory.md`），并以可复现构建为导向。

## 1. 先把“AI 输入法”拆成几类（否则会被营销词带跑）

1. **AI 作为候选生成器（文本补全/改写/续写）**
   - 特征：拿到用户“已输入的上下文”，给出 1~N 个建议；可能是“每键触发”或“按钮触发”
2. **AI 作为语音输入（ASR）**
   - 特征：把音频转文本，交给系统输入；对中文体验的关键是 VAD、纠错、标点、热词
3. **AI 作为工作流/Agent（跨 App 操作）**
   - 特征：不只是出文本，还会“执行动作”；这往往超出输入法的权限边界，风险最大

我认为 **(1) 与 (2) 最有现实意义**；(3) 更像“系统级助手”，输入法只是入口之一。

## 2. 我在 GitHub 看到的“AI 键盘”典型实现（可复用的设计点）

### 2.1 LLM 文本补全：按钮触发优于“每键触发”

本地仓库：`TODO/ime-research/repos/bosowski-keyboardgpt`

- 方案：Android IME + OpenAI LLM API（通过后端代理），用“按钮”生成建议，而不是每次按键都调用
- 作者在 README 里直说：**LLM 延迟太高，不适合替代本地 n-gram 做每键预测**；但“按按钮生成”仍然有价值

我的判断：
- **这条路线很真实**：只要你追求“毫秒级输入流”，LLM 就必须退到“慢路径”
- 真正的产品难点不在调用 API，而在：
  - 触发时机（何时建议、何时闭嘴）
  - 建议的“可控、可撤销、可解释”
  - 成本（token 费用）与隐私（输入内容天然敏感）

### 2.2 语音输入：离线 Whisper IME 的工程权衡

本地仓库：`TODO/ime-research/repos/woheller69-whisperime`

- 方案：Whisper 语音识别，既可作为 IME，也可作为系统级 voice input（RecognitionService）
- 隐私路线：首次下载模型需要网络权限，之后可离线

我的判断：
- ASR 比 LLM 文本补全更“闭环”：语音->文本->输入，用户预期清晰、容错空间更大
- 语音输入的瓶颈通常是：
  - VAD（何时开始/结束）
  - 低延迟与低功耗
  - 标点/断句/同音纠错（中文尤甚）

### 2.3 “系统级语音转输入”小工具（Windows/Android 都可参考）

本地仓库：`TODO/ime-research/repos/whisper-to-input`

- 更像“把识别结果注入到输入焦点”的工具链，而不是完整 IME
- 价值：验证“语音输入”这条能力在桌面端的可用性（尤其 Windows）

## 3. AI 输入法最大的挑战（我自己的主见）

### 3.1 最大矛盾：输入法是“快系统”，大模型是“慢系统”

- 输入法的手感底线：按键到候选刷新通常在 **10~50ms 量级**才不显“卡”
- 大模型生成：哪怕是云端小模型，也常见 **秒级**；端侧模型还受限于算力、电量、散热

因此必须设计“双路径”：
- **快路径**：本地规则/词典/小模型，保证每键反馈
- **慢路径**：显式触发（按钮/选中文本/句末），异步给结果，用户可一键采用/撤销

### 3.2 第二大挑战：信任与合规（输入法天然敏感）

- Android/Windows 输入法对用户来说是“系统级权限”：它能看到你输入的任何内容
- 一旦接入云端 LLM，用户会天然担心“泄露/训练/合规”

我的建议：
- 先做“完全离线可用”的输入法本体，再把 AI 做成**可选插件**（默认关闭）
- AI 能力从“可撤销”的开始：改写、润色、翻译、摘要，而不是自动发消息/自动执行

### 3.3 第三大挑战：交互协议（候选、光标、上下文的边界）

输入法不是一个普通文本框应用，它面对的是：
- 光标位置可能变化
- 应用可能分段接收 composing text / committed text
- 不同 App/输入框实现差异巨大（尤其 Android 各厂商 ROM，Windows 各类控件）

如果 AI 生成的文本与用户输入状态不同步，就会出现：
- 串字、丢字、回退异常
- 候选窗位置错乱
- “用户已经改了，AI 还在输出旧上下文”

## 4. 结论：有没有“现成方案”？怎么用才靠谱？

我认为现成的开源项目**能提供方法论与可跑的 demo**，但你很难“直接拿来当主力生产输入法”：
- LLM 补全类：更多是探索性质，关键问题是成本/隐私/延迟/可控性
- Whisper/语音类：更接近可产品化，但要面对模型体积、端侧性能、分发政策等现实

如果你的主目标是“做一个 Windows + Android 都能用、可长期维护的输入法”：
1. 主线建议仍是 **成熟引擎（Rime/Fcitx）+ 多前端**（先把输入体验做稳）
2. AI 先做成 **慢路径能力**：按钮触发、选中触发、句末触发
3. 最终再考虑端侧小模型（本地候选/纠错）与云端大模型（改写/总结）组合
