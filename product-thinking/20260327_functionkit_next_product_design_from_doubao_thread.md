# 从 doubao-thread 提炼的下一阶段产品设计：AI 输入法 Function Kit 平台

> 来源材料：`product-thinking/doubao-thread-wce998b8e58c59ca0.md`（PDF 文本提取 + 多轮对话整理）
>
> 目的：把“能跑起来的功能件运行时 + AI 能力”推进到“用户会长期用、开发者愿意做、生态可持续”的产品设计与落地路线。

## 0. 一句话定位

不是“带 AI 的输入法”，而是 **以输入法为无摩擦入口的插件化表达增强层（Function Kit Platform）**：在任何输入框里，基于当前上下文，用 1-2 步完成原本需要复制粘贴和多应用切换才能完成的动作。

## 1. 从材料里抽出的「产品必答题」

材料反复强调：创新点不在“能实现”，而在“能高频、自然地被使用”。因此下一阶段设计必须先回答并固化成产品约束：

1. **不可替代性来自哪里**：是输入法载体（表达链路第一触点、发送前最后一米、全场景覆盖）还是功能件生态？如果大厂也做功能件市场，你的护城河是什么？
2. **第一决策理由是什么**：用户愿意替换掉用了多年的输入法，必须为了一个尖刀级的功能而来，而不是“功能很多”。
3. **用户心智取舍**：用户把它当“更好用的输入法”，还是“装在输入法里的万能工具平台”？两者对应的交互、默认配置、宣发策略完全不同。

结论：下一阶段先把产品定义为 **“更好用的输入法 + 少量尖刀功能件（开箱即用）”**，平台化能力隐藏在后面逐步打开，避免心智失焦。

## 1.5 市场洞察（来自 PDF 部分的“AI 输入法/语音输入法”调研）

材料给出的市场结构与机会点，对你下一阶段的产品取舍有直接指导意义：

- 市场存在三类路线：移动端大厂派（低延迟、强入口）、桌面 SaaS 派（高质量输出、订阅）、小成本极客派（本地优先、隐私、需要自配 API）。
- 输入法的入口价值被重估，但隐私与合规风险同步放大（输入法天然接触最敏感数据，监管与用户疑虑都更强）。
- 对新入局者的机会点更偏“差异化”而非“全能”：垂直场景深耕、跨平台一致性、隐私优先、离线/弱网体验。

对你的直接结论：不要把下一阶段目标定为“在通用场景里和巨头拼能力”，而应把差异化放在 **输入链路的独占优势**（发送前、剪贴板闭环、场景识别、零切换）与 **插件化生态**（可扩展、可控）上。

## 2. 两条红线 + 三条硬规范（材料中最强共识）

### 2.1 两条红线

1. **不要把“功能件商店”当日常入口**：商店只负责“发现/安装/管理”，日常调用必须在输入流里 1 步完成（/ 指令、工具栏、复制后入口、场景入口）。
2. **默认极简，按需扩展**：默认只预装 1-2 个尖刀功能件；其余都由用户自主启用/固定/授权。避免“功能堆砌导致选择负担”。

### 2.2 「1-1-1」强制规范（所有功能件都必须服从）

1. **1 步唤起**：/ 指令、工具栏、复制后入口、场景入口，任选其一；禁止多级菜单深跳。
2. **1 步执行**：打开即给最优默认方案；用户点击一次即可出结果；复杂设置放到“高级”里，默认不出现。
3. **1 步返回**：执行完能一键回到输入界面；不制造多余页面与弹窗。

## 3. “浏览器插件式”平台必须补齐的四块基础设施

材料用浏览器插件成功经验倒推：想做功能件生态，必须先把“低门槛开发 + 可发现分发 + 权限透明 + 可持续激励”这四件事设计清楚。

### 3.1 标准化 API 与低门槛 SDK

目标：普通前端开发者 1 天内能做出可用功能件，而不是只有极客能玩明白。

设计要点：

- API 少而强：围绕“上下文读取、AI 调用、上屏、存储、任务”五类核心能力设计稳定接口。
- 交互模型简单：以“面板 UI + 宿主显式提交（insert/replace/apply）”为主，避免复杂双向状态机外泄给开发者。
- 调试工具可见：必须有可观测的版本号、权限、消息收发、错误堆栈与重放能力（否则开发者无法迭代）。

### 3.2 功能件商店（发现/分发）

必须提供：分类、搜索、排行榜、精选、最近使用。不要让用户为了“用功能”而进商店。

### 3.3 权限体系与审核

材料强调输入法的权限敏感度远高于浏览器：信任是生命线。

最低要求：

- 权限最小必要：功能件申请的权限必须透明可见，用户可随时撤销。
- 本地优先：尽可能多的功能件纯本地运行，减少隐私顾虑与联网依赖。
- 审核与熔断：上架审核 + 举报下架 + 紧急熔断（防止恶意功能件）。

### 3.4 变现与激励

必须在设计层面预留：订阅/买断/赞助/企业版功能件等合规变现路径，并给开发者稳定分成预期，否则生态会死。

## 4. 能力模型：Function Kit 需要的“上下文、触发、权限”

材料把输入法功能件与浏览器插件类比，并给出“输入法能拿到什么上下文”的分层思路。下一阶段应把这些分层固化成产品/SDK 的主模型。

### 4.1 上下文（Context）分层

1. **核心静态内容上下文**：输入框全量文本 + 选区（仅限输入框内）、光标附近文本。
2. **剪贴板上下文**：用户明确复制过的内容（高频、意图明确、最适合做“黄金触发点”）。
3. **宿主场景上下文**：宿主应用（包名/进程）、输入框类型（单行/多行/密码/数字/邮箱等）。
4. **系统环境上下文**：联网状态、设备形态（触屏/桌面）、平台能力差异。
5. **用户偏好上下文**：常用短语、常用功能件、个人写作风格等（本地存储）。
6. **动态输入时序上下文（可选）**：回退/删除/停顿/重写等“纠结信号”，用于在极克制的前提下触发辅助。

关键约束（来自材料的现实校验）：

- “selection” 只可靠于 **输入框内**（不要误解为能读到 app 页面里的任意选区）。
- 最通用的跨应用触发是 **clipboard**，而不是正文 selection。

### 4.2 触发（Trigger）优先级

材料明确：输入法的用户容错极低，**默认只做用户主动触发**，动态监听只能作为白名单可选能力。

建议的触发优先级（从稳到冒险）：

1. 显式触发：工具栏按钮、/ 指令、面板内按钮。
2. 复制触发：复制后 5 秒入口（提示 chip/通知），用户点击才进入处理。
3. 提交拦截：发送/提交前最后一道防线（高价值但必须极稳，且强可解释）。
4. 动态监听：纠结点识别、预输入状态预判（默认关闭，强约束频率与打扰度）。

### 4.3 权限（Permissions）建议基线

以“最小必要”原则划分，SDK/宿主只暴露功能件需要的抽象能力，而不是原始系统 API：

- `context.read`：读输入框内容、光标上下文、宿主场景信息
- `clipboard.read` / `clipboard.write`（建议）：读写剪贴板（写建议由宿主提供，避免功能件直接碰系统剪贴板 API）
- `input.insert` / `input.replace`：显式上屏（必须由宿主提交，避免功能件暗改）
- `ai.chat`：统一 AI 调用入口（敏感信息由宿主集中管理）
- `network.fetch`（谨慎）：如果提供，必须宿主代理并带权限与域名/速率控制（避免“随便 fetch”变成安全黑洞）
- `task.create` / `task.query`：所有异步都封装为 task（见下一节）

## 5. AI 设计：承认“慢”，用任务与交互协议解决“慢”

材料对“实时性”的最终结论是：大模型调用天然秒级，真正难点不是让 AI 变快，而是设计 **“实时输入流” 与 “异步 AI 任务” 的协作协议**。

### 5.1 两层 AI（快脑/慢脑）在输入法中的落地

- 快层（<200ms）：用于触发决策、轻量意图识别、基本风险词库匹配（可规则/小模型/本地模型）。
- 慢层（2-10s）：云端 LLM 生成/推理/改写/总结等，必须异步。

### 5.2 “人机异步协作协议”必须具备的 UI 语义

1. **不阻塞输入**：AI 在跑时用户仍能继续打字。
2. **可取消/可忽略**：用户不想要就一键关，不要“追着用户跑”。
3. **结果投递低侵入**：以卡片/按钮/候选建议形式出现；用户一键插入/替换；不要弹窗打断心流。
4. **结果与触发上下文绑定**：用户在 AI 运行期间修改了输入内容，必须让功能件能识别“上下文已变更”并提示是否重算。

### 5.3 统一任务（Task）体系

材料多次指向：异步不可避免，因此“面向用户的异步”应标准化：

- 功能件发起 AI 或外部操作，一律创建 task。
- 输入法提供一个统一入口查看 task：运行中、成功、失败、可重试、可复制结果、可上屏。
- task 记录最小可解释日志：触发原因、使用权限、访问了哪些上下文、结果来源（本地/云端）。

## 6. 杀手级场景的产品化：从“想法清单”到“可落地组合拳”

材料给了大量场景灵感，但下一阶段要做的是收敛成“尖刀功能件 + 触发链路 + 权限模型 + 指标”的可落地方案。

### 6.1 场景筛选标准（强制）

1. 输入法独有：必须利用“表达链路/发送前/剪贴板闭环/宿主场景识别”之一，否则独立 app 可以吊打。
2. 高频：每天能用多次，或者一用就强传播。
3. 低风险：不依赖灰产、不依赖脆弱适配、不碰敏感权限红线。
4. 1-1-1：必须能做到 1 步唤起、1 步执行、1 步返回。

### 6.2 推荐优先级最高的 3 个“尖刀功能件”

1. **复制内容处理（Clipboard Hub）**
   - 触发：复制后 5 秒入口（用户点击）
   - 能力：翻译/总结/清格式/结构化提取/脱敏/润色
   - 闭环：结果一键 insert/replace 回当前输入框
2. **发送前最后一道防线（Send Intercept）**
   - 触发：发送动作/提交动作
   - 能力：合规风险提示 + 情绪冷静 + 敏感信息检查（手机号/证件/金额单位等）
   - 约束：必须强可解释、可一键忽略、默认克制
3. **宿主场景适配（App-aware Rewrite）**
   - 触发：用户主动点功能件或 / 指令
   - 能力：同一段内容一键适配“微信/邮件/小红书/抖音口播稿/公文”等不同表达场景（语气、段落、格式、长度）

### 6.3 动态监听的“可用而不恶心”的边界

材料认为动态监听不是不能做，但必须只做“主动触发做不到”的真痛点，并且极克制：

- 纠结点识别：反复修改/长停顿/整段重写，只给一个极小提示，用户点击才展开。
- 预输入辅助：基于拼音/预编辑状态做“注音释义/英文拼写/数字转换”的轻量提示。

默认策略：**默认关闭 + 白名单 + 频率上限 + 明确可解释**。

## 7. 功能件联动：从“孤岛”到“流程”

材料把“功能件孤立”列为典型重大缺陷，并给出三层联动思路。下一阶段建议把联动设计成平台能力，而不是每个功能件各自实现。

1. **原子能力融合**：把能力拆成可组合的原子动作，允许 `/小红书` 这种复合指令串联多个动作。
2. **执行后推荐下一步**：功能件完成后，给 1-2 个下一步建议（可关）。
3. **用户自定义工作流**：工作流编辑器（后置，先验证 1 与 2）。

## 8. 分阶段路线图（材料建议的“克制式”推进法）

### 阶段 1：MVP（把一个点打穿打透）

目标：用一个高频动作闭环获得信任与留存。

- 选择一个黄金触发点：优先 clipboard；其次输入框内 selection。
- 做 3 个功能做到极致可靠：翻译/清格式/结构化提取（或你团队最有把握的三项）。
- 统一交互：入口 -> 处理 -> insert/replace -> 任务记录。

成功指标建议：

- 平均每日触发次数（>3/天/人）
- 7 日留存（>50% 作为早期合理目标）
- 触发到出结果延迟（本地 <200ms，AI 异步可接受但必须不阻塞）

### 阶段 2：场景化与联动（把“好用”变成“离不开”）

- 宿主 app/输入框类型适配（社交/办公/编程）
- 执行后推荐下一步
- 权限面板与隐私仪表盘（让用户敢用）

### 阶段 3：平台化与生态（把“一个产品”变成“一个市场”）

- SDK/模板/调试工具完善
- 商店与审核、评分、举报、下架
- 变现与分成机制

## 9. 与跨平台技术路线相关的产品约束（来自 PDF 提取部分）

材料对“跨平台输入法”给出的结论是：从零做难度极高，基于成熟引擎/框架可显著降低风险（例如 Rime/librime、Fcitx 等）。

对产品设计的直接含义：

- **不要用产品设计去倒逼底层重造轮子**：优先把“功能件平台能力”做成跨端一致的上层（Web UI + runtime + host 能力抽象），底层 IME 引擎按平台选成熟方案。
- **一致性优先于“全能”**：跨端一致的触发、权限、任务、上屏语义，比“某端做出更多黑科技”更重要。

## 10. 下一步需要你（产品 owner）拍板的决策清单

从材料提炼出来，最影响后续路线的 5 个“必须拍板”的产品决策：

1. MVP 的第一尖刀功能件是哪一个？（建议从 clipboard 闭环选一个最强的）
2. 动态监听默认是否开启？（建议默认关闭）
3. `network.fetch` 是否提供给功能件？如果提供，域名/速率/透传策略是什么？
4. AI 配置是否全局托管？（建议托管，功能件不接触密钥）
5. 生态启动策略：先自研一批官方功能件，还是优先开放第三方？（建议先自研 3-5 个打样，形成规范）
